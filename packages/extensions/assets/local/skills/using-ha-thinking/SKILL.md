# using-ha-thinking

## 규율 (단 하나)

**한 층에 쓴 문장은 바로 아래 층 용어 없이 닫혀야 한다.**

지키면 위·아래 독자 모두 자기 수준에서 읽을 수 있다.
깨지면 리뷰·재진입이 아래 층에 묶이고, 위 층의 의도는 복원 불가.

## 층이란

하나의 아이디어가 실제 실행이 되기까지 거치는 단계. 몇 개 필요한지는 아이디어와 그 시점의 도구가 함께 결정한다.

- 프레임워크가 두꺼운 영역 — 층이 적다 (아이디어 → 한 줄 코드)
- 새로 만들 게 많은 영역 — 층이 많다 (아이디어 → 설계 → 아키텍처 → 모듈 → 함수 → 라인)
- 같은 아이디어라도 시간이 지나 프레임워크가 아래 층을 흡수하면 층 수가 준다

숫자 라벨(L1, L7 등)은 쓰지 않는다. 층이 고정 자원이라는 오해를 만들고, 매 문제마다 다른 층 수가 필요하다는 사실을 가린다.

## Process

**1. 층 수 정하기.**
"이 아이디어를 실행 가능한 단위로 내리려면 몇 번 쪼개야 하는가?" N번이면 N층.
모호하면 N을 줄인다. 나중에 쪼개는 게 미리 부풀리는 것보다 싸다.

**2. 각 층의 언어 정하기.**
층마다 독자가 있다. 위로 갈수록 "왜", 아래로 갈수록 "어떻게". 중간 층은 바로 위 질문에 답하고, 바로 아래로 넘길 질문을 만든다. 두 층 아래 용어는 쓰지 않는다.

**3. 경계 명시.**
경계 없이 한 문서에 다층 금지. 경계는 별도 파일·이슈·PR, 또는 같은 문서 안에서 `---` + 새 헤더.
경계 하나당 층 하나.

**4. 움직임.**
- "왜 이걸 하는지 모르겠다" → 한 층 위
- "뭘 해야 할지 모르겠다" → 한 층 아래

## 층이 부패하는 증상

- 기획 문서에 함수명·파일 경로가 나옴
- 한 문단 안에서 "왜"와 "어떻게"가 섞임
- 리뷰가 구현 디테일에서 멈춤, 위 층 의도가 검토되지 않음
- 세션이 끊기면 재진입 불가 (아래 층 맥락 없이 위 층이 해석 안 됨)

## Anti-patterns

- 모든 문제에 같은 층 수 강제
- 숫자 라벨로 층을 고정화
- 위 층 본문에 함수명·파일 경로·ENV 변수·코드 블록
- 경계 없이 한 문서에 다층
- 아래 층으로 내려갈 출구가 없는 위 층 (추상만 있고 구체로 갈 길이 없음)
