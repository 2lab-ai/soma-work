# ztrace — Scenario-Driven Callstack Trace

시나리오를 입력으로 받아 콜스택 수준으로 추적하고, "왜 작동하는가"까지 설명한다.

## Core Principles

1. **시나리오가 입력이다** — "코드 분석해줘"가 아니라 "이 행위가 어떻게 동작하는지 추적해줘". 시나리오가 없으면 에이전트가 먼저 도출하고 유저에게 확인받는다.
2. **깊이는 콜스택이 기본값이다** — 메서드 이름만 나열하는 게 아니라 `[파일:L42]` 수준까지 내려간다. SQL 쿼리 원문, HTTP 호출 URL, 조건 분기까지 전부.
3. **"왜"가 "뭘"보다 중요하다** — 코드 나열은 IDE가 한다. 에이전트의 가치는 "이 패턴이 동시성/분산 환경에서 왜 안전한가"를 설명하는 데 있다. 상태 머신 종합은 이 "왜"의 최종 결정체.

## Input

```
{대상 기능}의 주요 워크플로우를 시나리오별로 콜스택 수준 추적해줘

시나리오:
• {유저 행위 1 — 트리거 → 기대 결과}
• {유저 행위 2 — 트리거 → 기대 결과}
• ...
```

## Process

### Phase 0: 시나리오 도출

**유저가 시나리오를 제공하지 않은 경우:**
1. 대상 기능의 엔트리포인트 탐색 (Controller, Worker, EventHandler)
2. 유저 관점의 핵심 시나리오 3~6개 자동 도출
3. `local:UIAskUserQuestion`으로 시나리오 확인 받기

**유저가 시나리오를 제공한 경우:**
→ Phase 1로 직행

### Phase 1: 소스 전수 탐색

1. 대상 기능의 모든 관련 파일 식별 (Explore agent)
   - Controller → Service → DB Service → Entity
   - Worker → Background loop → External API call
   - Interface → Implementation → DI registration
2. 모든 파일의 메서드 시그니처 + 라인 번호 수집
3. 외부 호출 맵 구성 (메서드 A → 메서드 B 체인)

### Phase 2: 시나리오별 콜스택 추적

각 시나리오에 대해:

1. **트리거 식별**: 어디서 시작되는가 (API call / Timer / Event)
2. **콜스택 작성**: 들여쓰기 트리 형식으로 전체 호출 체인
   - 각 호출에 `[파일:라인]` 표기
   - 조건 분기마다 `[조건] → 결과` 표기
   - SQL 쿼리는 원문 포함
   - 외부 API 호출은 HTTP method + URL 포함
3. **에러 핸들링 경로**: 각 단계에서 실패 시 어떻게 처리되는가
4. **설계 의도 ("왜 작동하는가")**:
   - 왜 이 패턴을 선택했는가
   - 동시성/분산 환경에서 어떻게 안전한가
   - 어떤 엣지 케이스를 방어하는가

**출력 형식:**

```
시나리오 N: {제목}
트리거: {트리거 설명}
상태 전이: {시작 상태} → {중간 상태} → {최종 상태}

{콜스택 트리 — 파일:라인 참조 포함}

왜 작동하는가:
1. {설계 원칙 1}: {구현 방법}
2. {설계 원칙 2}: {구현 방법}
```

### Phase 3: 상태 머신 종합

1. 모든 시나리오에서 관찰된 상태 전이를 수집
2. ASCII 상태 머신 다이어그램 생성
3. 전이 테이블: 전이 / 트리거 / 코드 경로
4. 불변식(invariant) 목록: 시스템이 보장하는 것들

### Phase 4: 설계 원칙 요약

| 원칙 | 구현 |
|------|------|
| {원칙 1} | {어떻게 구현되었는가} |
| {원칙 2} | {어떻게 구현되었는가} |

## Quality Checklist

- [ ] 모든 콜스택에 `[파일:라인]` 참조 있음
- [ ] 조건 분기마다 `[조건] → 결과` 명시
- [ ] SQL 쿼리 원문 포함 (ORM 뒤에 숨지 않음)
- [ ] 외부 API 호출 HTTP method + URL 포함
- [ ] 에러 핸들링 경로 빠짐없이 추적
- [ ] 각 시나리오 끝에 "왜 작동하는가" 섹션 있음
- [ ] 상태 머신 다이어그램 + 전이 테이블 있음
- [ ] 설계 원칙 요약 테이블 있음
