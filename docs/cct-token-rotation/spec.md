# CCT Token Rotation Spec — AuthKey v2

> Updated: 2026-04-19 — AuthKey v2 foundation (#575 PR-A). Supersedes the
> 2026-04-18 Wave 5 spec, which described the v1 `TokenSlot` union. The
> v1 spec is retained as `trace.md` (with a DEPRECATED banner) for
> history; this document is the authoritative reference for anything
> landing on top of `main@HEAD` after PR-A.

## Overview

The CCT (Claude Code Token) subsystem owns a persistent, cross-process
pool of Anthropic credentials. Each credential lives in a named **slot**
described by the `AuthKey` union. The manager:

- Selects one slot as "active" and surfaces its access token through a
  per-lease env object (via `buildQueryEnv`). Spawned `claude` CLI
  processes inherit `CLAUDE_CODE_OAUTH_TOKEN` from that env, not from
  `process.env`.
- Rotates to the next healthy slot on rate-limit (detected from response
  headers or error strings) or manual operator action.
- Refreshes `oauthAttachment` credentials proactively ≥ 7 h before
  expiry, deduped in-process and serialised across processes via a
  single lockfile.
- Polls the usage endpoint per slot with a 2→5→10→15 min backoff ladder
  so the `/z cct` card surfaces accurate 5h / 7d utilisation without
  saturating Anthropic's quota endpoint.
- Evicts revoked / quarantined slots and drains in-flight leases cleanly.

Code entry points:

- `src/auth/auth-key.ts` — the `AuthKey` union, `OAuthAttachment`, and
  helper type guards.
- `src/cct-store/store.ts` + `types.ts` — durable slot storage (schema
  v2, revision CAS, fsync chain, `cct-store.lock`).
- `src/cct-store/migrate-v2.ts` — pure v1 → v2 snapshot migrator.
- `src/token-manager.ts` — the `TokenManager` class; owns lease
  lifecycle, rotation, proactive refresh, and usage fetch.
- `src/credentials-manager.ts` — lease contract (`SlotAuthLease`) and
  the `ensureActiveSlotAuth` wrapper used by every dispatcher.
- `src/auth/query-env-builder.ts` — per-lease env-object builder.
- `src/oauth/refresher.ts` / `usage.ts` / `scope-check.ts` /
  `header-parser.ts` — HTTP-layer helpers (targeted for PR-C island
  extraction).
- `src/slack/cct/builder.ts`, `views.ts`, `actions.ts` — Block Kit card
  & modal surface.
- `src/slack/commands/cct-handler.ts`, `src/slack/command-parser.ts` —
  text-command grammar (READ-ONLY for token mutation).

## AuthKey model (schema v2)

### Identity (`keyId` / `name`)

Every slot has an immutable `keyId` (ULID, generated by `ulid()` in
`TokenManager.addSlot`) and a mutable `name` (max 64 chars). All
internal maps, `Lease.ownerTag` suffixes, usage caches, and refresh
dedupe keys are keyed by `keyId` — renaming a slot does **not**
invalidate in-flight work. `name` is the only identifier surfaced in
Slack UI (Block Kit and text commands).

The v1 field name `slotId` was renamed to `keyId` as part of schema v2.
Legacy v1 snapshots on disk are migrated in-memory on `bootstrap()`.

### The 2-arm union (`AuthKey`)

`AuthKey` in `src/auth/auth-key.ts` is a tagged union on `kind`:

```typescript
export type AuthKey = ApiKeySlot | CctSlot;

export interface ApiKeySlot {
  kind: 'api_key';
  keyId: string;
  name: string;
  value: string;          // sk-ant-api03-…
  createdAt: string;
}

export type CctSlot = CctSlotWithSetup | CctSlotLegacyAttachmentOnly;

export interface CctSlotWithSetup {
  kind: 'cct';
  source: 'setup';
  keyId: string;
  name: string;
  setupToken: string;     // sk-ant-oat01-…
  createdAt: string;
  oauthAttachment?: OAuthAttachment;
}

export interface CctSlotLegacyAttachmentOnly {
  kind: 'cct';
  source: 'legacy-attachment';
  keyId: string;
  name: string;
  createdAt: string;
  oauthAttachment: OAuthAttachment;
}

export interface OAuthAttachment {
  accessToken: string;
  refreshToken: string;
  expiresAtMs: number;
  scopes: string[];
  rateLimitTier?: string;
  subscriptionType?: string;
  acknowledgedConsumerTosRisk: true;   // literal, required
}
```

**Why the split.** v1 lumped setup tokens and OAuth credentials into
sibling slot kinds. That made it impossible to represent a *seat-
provisioned* slot (setup token bound to a Max seat) that *also* carries
a refreshable OAuth bundle issued for that seat. v2 makes the setup
token the identity (`source: 'setup'`) and treats OAuth as an optional
attachment. The `legacy-attachment` source preserves v1 OAuth-only slots
so the migration is lossless.

**`api_key` is new.** PR-A makes the shape representable; the
acquisition path and Slack UI support land in PR-B.

### `SlotState`

Per-slot mutable state (keyed by `keyId` in the snapshot's `state` map,
unchanged by shape from v1 — only the key was renamed):

```typescript
export interface SlotState {
  rateLimitedAt?: string;
  rateLimitSource?: 'response_header' | 'error_string' | 'manual';
  cooldownUntil?: string;
  authState: 'healthy' | 'refresh_failed' | 'revoked';
  activeLeases: Lease[];
  tombstoned?: boolean;
  lastUsageFetchedAt?: string;
  nextUsageFetchAllowedAt?: string;
  usage?: UsageSnapshot;
}
```

Rotation eligibility is computed by `isEligible(state, nowMs)` in
`token-manager.ts` — tombstoned, revoked, refresh_failed, and
still-cooling slots are excluded.

## Persistence (schema v2)

### `cct-store.json` shape

A single file under the soma-work data dir (`defaultCctStorePath()`
resolves to `<envPaths.data>/cct-store.json`). Snapshot shape:

```typescript
export interface CctStoreSnapshot {
  version: 2;
  revision: number;
  registry: { activeKeyId?: string; slots: AuthKey[] };
  state: Record<string, SlotState>;
}
```

### v1 → v2 migration

`migrateV1ToV2(raw)` in `src/cct-store/migrate-v2.ts` is pure and
idempotent:

- `registry.activeSlotId` → `registry.activeKeyId`.
- `SetupTokenSlot { kind:'setup_token', slotId, value, … }` →
  `CctSlotWithSetup { kind:'cct', source:'setup', keyId, setupToken, … }`.
- `OAuthCredentialsSlot { kind:'oauth_credentials', slotId, credentials, …, acknowledgedConsumerTosRisk:true }`
  → `CctSlotLegacyAttachmentOnly { kind:'cct', source:'legacy-attachment', keyId, createdAt, oauthAttachment: { …credentials, acknowledgedConsumerTosRisk:true } }`.
- `state` keys preserved 1:1 (old `slotId === new keyId`).
- `version` bumped to `2`; `revision` preserved.

The migrated snapshot stays in memory until the next mutation triggers
a CAS write — a failed bootstrap cannot corrupt a healthy v1 disk file.

### Revision CAS

`CctStore.save(next)` fails with `CctStoreStaleWriteError` if
`next.revision !== current.revision + 1`. `CctStore.mutate(fn)` is the
correct all-purpose entry point: it loads the latest snapshot, runs
`fn` under the lock, bumps `revision`, and persists. Consumers that
race will each see a fresh `revision` before mutating.

### Durable fsync-chain writes

`CctStore.save` writes to a `*.tmp` sibling, `fsync`s that file, then
`rename()`s it over `cct-store.json`, then `fsync`s the parent
directory. Post-crash on-disk state is either the pre-image or the full
post-image.

### Locking

`cct-store.lock` is the **only** cross-process lock. `CctStore.withLock(fn)`
is a bounded spin on `O_EXCL` with a hand-off file. The TokenManager
uses it for every mutation. Refresh and usage-fetch dedupe use
**in-process** `Map<keyId, Promise<…>>` only; HTTP calls happen outside
the store lock so we never hold the lock across the network. Lock
ordering is strict: caller → store lock → done.

## Lease contract

`src/credentials-manager.ts` exposes `ensureActiveSlotAuth(tm, ownerTag, ttlMs?)`,
which is the **only** supported way for dispatchers to hold an auth
checkout:

```typescript
export interface SlotAuthLease {
  readonly keyId: string;
  readonly kind: AuthKey['kind'];       // 'api_key' | 'cct'
  readonly accessToken: string;          // renamed in PR-B — see trace-v2.md §10
  release(): Promise<void>;              // idempotent
  heartbeat(): Promise<void>;            // no-op after release
}
```

- For `kind === 'cct'` with `oauthAttachment`, `accessToken` is the
  value returned by `tm.getValidAccessToken(keyId)` — refreshed
  transparently if it's within the 7h buffer.
- For `kind === 'cct'` without an attachment, `accessToken` is the
  `setupToken` as-is.
- For `kind === 'api_key'`, `accessToken` is the API key value. (PR-A
  makes the shape representable; PR-B wires the acquisition path.)

Legacy `ensureValidCredentials(): Promise<{valid: boolean, error?: string}>`
is a thin wrapper that acquires, immediately releases, and returns a
boolean. Kept for non-dispatcher callers that only need a health check.

## Query env propagation

`src/auth/query-env-builder.ts::buildQueryEnv(lease)`:

```typescript
export function buildQueryEnv(lease: SlotAuthLease): { env: NodeJS.ProcessEnv } {
  return {
    env: {
      ...process.env,
      CLAUDE_CODE_OAUTH_TOKEN: lease.accessToken,
    },
  };
}
```

Contract:

- Never mutates `process.env`.
- Returns a fresh object per call — concurrent dispatches cannot collide.
- The returned env is handed to `@anthropic-ai/claude-agent-sdk` via
  `query({ env })`. The SDK forwards it verbatim to `child_process.spawn`.
- PR-B extends this to inject a per-`keyId` `CLAUDE_CONFIG_DIR` so
  concurrent queries using different slots cannot stomp each other's
  `~/.claude/config.json` state.

## OAuth refresh

### Endpoint

`POST https://platform.claude.com/v1/oauth/token` (see
`src/oauth/refresher.ts`). Body:
`{ grant_type: "refresh_token", refresh_token, client_id }`.

### Client ID constant

`CLAUDE_OAUTH_CLIENT_ID = '9d1c250a-e61b-44d9-88ed-5944d1962f5e'` — the
hard-coded client ID shipped with the Claude Code CLI. Exported by
`refresher.ts`.

### Data flow

`refreshClaudeCredentials(current: OAuthCredentials): Promise<OAuthCredentials>`:

- Input: the HTTP-DTO shape (`OAuthCredentials`) containing the current
  `refreshToken` and the `rateLimitTier` / `subscriptionType` to
  preserve across the refresh.
- Output: a fresh `OAuthCredentials` with `accessToken`, `refreshToken`
  (may equal input if the server did not rotate), `expiresAtMs =
  Date.now() + expires_in*1000`, and the preserved tier/subscription.

The `OAuthCredentials` DTO lives in `oauth/refresher.ts` — deliberately
decoupled from the persisted `OAuthAttachment`. `TokenManager.applyRefreshedCredentials`
is the seam that re-attaches `acknowledgedConsumerTosRisk: true` when
writing back to `slot.oauthAttachment`.

### Proactive 7h buffer

`TokenManager.getValidAccessToken(keyId)` checks
`expiresAtMs - Date.now() < REFRESH_BUFFER_MS` (7 h) and refreshes
*before* handing out an access token. This prevents a race where a
long-running stream inherits a token that expires mid-call.

### In-process dedupe + cross-process lock

`TokenManager.refreshAccessToken(slot)` stores the in-flight Promise in
`this.refreshInFlight: Map<keyId, Promise<string>>`. The first caller
races the HTTP POST; every other caller awaits the same Promise. The
write-back (`store.mutate(…)`) takes the store lock — two processes
refreshing simultaneously are tolerated but the CAS ensures only one
committed revision wins, and the other retries on the next access.

### Token rotation (refresh_token reuse semantics)

Anthropic's OAuth server may rotate `refresh_token` on every successful
refresh or return the old one unchanged. `refresher.ts` reuses the
input refresh token when the response omits it. `TokenManager`
persists the returned bundle atomically with the new access token. If
the next refresh returns 401, the manager sets
`authState = 'refresh_failed'`; 403 → `'revoked'`.

## Usage monitoring

### Endpoint

`GET https://api.anthropic.com/api/oauth/usage` (see
`src/oauth/usage.ts`, constant `CLAUDE_USAGE_URL`).

### `anthropic-beta` header

All usage requests include `anthropic-beta: oauth-2025-04-20` — the
endpoint is opt-in beta and will 404 without it.

### `UsageSnapshot` schema

Response is parsed into the typed `UsageSnapshot`:

```typescript
export interface UsageWindow {
  utilization: number;
  resetsAt: string;
}

export interface UsageSnapshot {
  fetchedAt: string;
  fiveHour?: UsageWindow;
  sevenDay?: UsageWindow;
  sevenDaySonnet?: UsageWindow;
}
```

`utilization` is a 0..1 float on the wire; renderers
(`src/slack/cct/builder.ts::toPct`, `cct-handler.ts::toPctInt`) scale
it to an integer percentage and also accept 0..100 for forward-compat.

### Per-slot `lastFetchedAt` + `nextUsageFetchAllowedAt`

`TokenManager.fetchAndStoreUsage(keyId)` persists
`state.lastUsageFetchedAt` (from `snapshot.fetchedAt`) and
`state.nextUsageFetchAllowedAt` (server success = +2 min; failure =
next step on the backoff ladder).

**`cct usage` gate**. The command handler (`cct-handler.ts::handleUsage`)
inspects the persisted slot and emits an "Usage API requires an OAuth
attachment" error when `slot.kind === 'cct'` has no `oauthAttachment`.
This replaces v1's `kind === 'setup_token'` check — now that `cct` is
one kind covering both variants, we discriminate on the attachment's
presence instead of on the slot kind.

### Backoff ladder: 2 → 5 → 10 → 15 (cap) minutes

`nextUsageBackoffMs(prev)` in `oauth/usage.ts` walks
`[2m, 5m, 10m, 15m]`. On every non-2xx, the manager calls
`applyUsageFailureBackoff(keyId)` which reads the current remaining
backoff and advances to the next step (capping at 15 min).

## Rate-limit detection

### Primary: response headers

`src/oauth/header-parser.ts::parseRateLimitHeaders` inspects
`anthropic-ratelimit-unified-5h-remaining`,
`anthropic-ratelimit-unified-7d-remaining`, and the matching `reset`
headers. When `remaining === 0`, the stream executor calls
`TokenManager.rotateOnRateLimit(reason, { source: 'response_header', … })`.

### Fallback: error string match

When headers are absent (older server build, network edge),
`parseCooldownTime(message)` parses `"resets 7pm"` /
`"resets Apr 7, 7pm (Asia/Seoul)"` from the error body and rotates with
`source: 'error_string'`.

### `rateLimitSource` field on SlotState

The detected source (`response_header` | `error_string` | `manual`) is
persisted on `SlotState.rateLimitSource` so the Slack card can render a
provenance badge (see `buildSlotRow` in `src/slack/cct/builder.ts`).

## Rotation + Lease

### `pickNextHealthy` (exclude tombstoned / revoked / cooling / refresh_failed)

`TokenManager.rotateToNext` and `rotateOnRateLimit` both walk
`snap.registry.slots` starting at `currentIndex + 1`, skipping any
slot where `isEligible(state, now)` is false. If none is eligible they
return `null` and leave the active pointer untouched.

### Lease lifecycle

- `acquireLease(ownerTag, ttlMs = 15m)` — appends a `Lease` to the
  active slot's `activeLeases`. If the active slot is ineligible,
  rotates first.
- `heartbeatLease(leaseId)` — extends `expiresAt` by the original TTL.
  Throws on expired / unknown leases.
- `releaseLease(leaseId)` — removes from `activeLeases`.
- `reapExpiredLeases()` — purges expired leases and fully removes
  tombstoned slots with zero leases. Invoked by the reaper timer every
  `DEFAULT_REAPER_INTERVAL_MS` (30 s).

### Tombstone + drain removal

`removeSlot(keyId)` returns `{removed: true}` when the slot is
leaseless, else `{removed: false, pendingDrain: true}` with
`state.tombstoned = true`. The reaper finishes the delete once all
leases have drained or expired.

## `authState`

- `healthy` — default; eligible for rotation.
- `refresh_failed` — last refresh attempt returned 401. Manager stops
  handing out tokens from this slot; operator must replace it.
  Surfaced as a yellow-circle badge on the card.
- `revoked` — 403 from refresh or 403 from usage. The credential is
  considered invalidated and the slot is quarantined.

`markAuthState(keyId, state)` is the sole mutator; called from
`refreshAccessToken`'s catch block and from `fetchAndStoreUsage`'s 403
branch.

## Slack UI

### Card blocks (`builder.buildCctCardBlocks`)

`src/slack/cct/builder.ts` renders the header + one section/context
pair per slot + the action row (`Next / Add / Remove / Rename`) + a
set-active selector when `slots.length > 1`. `buildSlotRow` pulls
`SlotState` and emits an authState badge, a rate-limit timestamp
(via `formatRateLimitedAt`), 5h / 7d utilisation, cooldown hint,
tombstone marker, and lease count.

### Add modal

`buildAddSlotModal(selectedKind: AddSlotFormKind)` — split fields
(name input, kind radio, conditional value input). The kind radio has
`dispatch_action: true` so flipping `setup_token ↔ oauth_credentials`
re-renders via `views.update` with stable `block_id` / `action_id`
keys so Slack preserves the user's typed values. `oauth_credentials`
adds a ToS-risk ack checkbox (`CCT_BLOCK_IDS.add_tos_ack`).

**UI kind vs. persisted kind.** `AddSlotFormKind = 'setup_token' |
'oauth_credentials'` is a UI-only discriminant used by the Slack
radio. The submit handler maps it to the persisted AuthKey shape:

- `setup_token` → `CctSlotWithSetup { kind: 'cct', source: 'setup', setupToken: value }`.
- `oauth_credentials` → `CctSlotLegacyAttachmentOnly { kind: 'cct', source: 'legacy-attachment', oauthAttachment: {...} }`.

The radio *could* surface `api_key` in a follow-on PR; today it does
not, so `AddSlotInput` is narrower than `AuthKey`.

### Remove modal

`buildRemoveSlotModal(slot: AuthKey, hasActiveLeases: boolean)` —
warns on active leases (slot will be tombstoned and drained).
`private_metadata` carries the `keyId` through `view_submission`.

### Rename modal

`buildRenameSlotModal(slot: AuthKey)` — single `plain_text_input`
seeded with the current name. `private_metadata` carries `keyId`.

### `block_id` / `action_id` stability across `views.update`

All modal field keys live in `src/slack/cct/views.ts` as frozen
constants (`CCT_BLOCK_IDS`, `CCT_ACTION_IDS`, `CCT_VIEW_IDS`). The
kind-radio flip re-uses the same keys so `state.values` survives the
update.

### 3s ack budget, `response_action:errors` keyed by `block_id`

`src/slack/cct/actions.ts` acks within 3 s and returns
`response_action: 'errors'` with messages keyed by `block_id` on
validation failure — Slack highlights the offending field inline
rather than closing the modal.

## Timestamp format

Canonical rate-limit timestamp format:

```
2026-04-18 13:42 KST / 03:42Z (5m ago)
```

Produced by `formatRateLimitedAt` in `src/util/format-rate-limited-at.ts`:
local wall clock in the user's IANA timezone (default `Asia/Seoul`),
UTC `HH:mmZ`, and a relative suffix (`now` / `Ns / Nm / Nh / Nd ago`).

Used by:

- `src/slack/cct/builder.ts::buildSlotRow` — per-slot context line.
- `src/slack/commands/cct-handler.ts::buildStatusTextFallback` — the
  plain-text fallback surfaced by `/z cct` when Block Kit is
  suppressed.

## Text command grammar (READ-ONLY)

Parsed by `src/slack/command-parser.ts::parseCctCommand`:

```
/z cct                  — show the Block Kit card
/z cct set <name>       — set the active slot
/z cct next             — rotate to next eligible slot
/z cct usage [<name>]   — show 5h / 7d usage snapshot for a slot
```

Forbidden via text (handler emits an error pointing to the card):

```
/z cct add …            — open the Add modal from the /z cct card
/z cct rm … | remove …  — open the Remove modal from the /z cct card
```

Rationale: split-field entry + ToS-risk ack + scope validation + modal
dispatch semantics are too rich to encode in Slack text; the card is
the only supported mutation surface.

## ToS risk framing

Consumer Claude subscriptions grant tokens via the `claudeAiOauth`
flow for the CLI's own use. Programmatic reuse of these tokens for a
bot arguably violates Anthropic's consumer ToS. Every
`OAuthAttachment` therefore carries the literal
`acknowledgedConsumerTosRisk: true` — enforced both by the Add modal
(checkbox) and by the `OAuthAttachment` type. Setup-only cct slots
(`source: 'setup'`, no attachment) are issued for programmatic use and
do not require the ack.

## Secret redaction

`src/logger.ts::redactAnthropicSecrets` replaces matches of
`/sk-ant-(?:oat01|ort01|api03|admin01)-[A-Za-z0-9_-]+/g` with
`sk-ant-…-REDACTED` across every log path. `installConsoleRedaction`
wraps `console.{log,info,warn,error,debug}` so `console.*` callers
also get filtered. Used in `TokenManager.applyToken`, `addSlot`, and
`refreshAccessToken` logging.

## Migration machinery

Two migrators cooperate on boot:

- `src/cct-store/migrate-v2.ts` — v1 → v2 snapshot migration (shape
  change, pure, idempotent). Applied unconditionally when
  `version === 1` is read off disk.
- `src/cct-store/migrate.ts` — legacy `<dir>/token-cooldowns.json`
  → `SlotState.cooldownUntil` merger. Writes
  `token-cooldowns.json.migrated.<ISO>` so it's audit-safe and only
  runs once.

`TokenManager.init` captures legacy entries *before* the migrator
renames the file so cooldowns can be re-applied after env-based slot
seeding.

## Backward compatibility

- `CLAUDE_CODE_OAUTH_TOKEN` — read by the spawned `claude` CLI from
  the per-query env crafted by `buildQueryEnv`. The global
  `process.env.CLAUDE_CODE_OAUTH_TOKEN` is not written to; callers
  that need it must opt in explicitly (legacy harness path).
- `CLAUDE_CODE_OAUTH_TOKEN_LIST` — comma-separated list seeded into
  `source: 'setup'` cct slots on first boot when the registry is
  empty. `name:value` and `name=value` forms both accepted; missing
  name falls back to `cct<N>`.
- `SOMA_CCT_DISABLE_ENV_SEED=true` — opt-out for test harnesses /
  containers that manage slots by other means.
- Legacy `/z cct set <name>` and `/z cct next` text commands still
  work unchanged — the slot model is invisible to operators who only
  ever use the text grammar.
- `ensureValidCredentials()` — legacy health-check wrapper. New code
  should call `ensureActiveSlotAuth(...)` directly.
