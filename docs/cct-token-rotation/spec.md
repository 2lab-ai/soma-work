# CCT Token Rotation Spec

> Updated: 2026-04-18 — Wave 5 of the slot-model overhaul (#569). Supersedes
> the original 2026-03-05 spec, which described the single-process, env-var
> based `TokenManager`.

## Overview

The CCT (Claude Code Token) subsystem owns a persistent, cross-process pool
of Claude credentials. Each credential lives in a named **slot** and can be
either a legacy `setup_token` (`sk-ant-oat01-…`) or a full
`oauth_credentials` bundle (access token + refresh token + scopes). The
manager:

- Selects one slot as "active" and mirrors its access token to
  `process.env.CLAUDE_CODE_OAUTH_TOKEN` so spawned `claude` CLI processes
  inherit it transparently.
- Rotates to the next healthy slot on rate-limit (detected from response
  headers or error strings) or manual operator action.
- Refreshes `oauth_credentials` proactively ≥ 7 h before expiry, deduped
  in-process and serialised across processes via a single lockfile.
- Polls the usage endpoint per slot with a 2→5→10→15 min backoff ladder so
  the `/z cct` card surfaces accurate 5h / 7d utilisation without
  saturating Anthropic's quota endpoint.
- Evicts revoked / quarantined slots and drains in-flight leases cleanly.

Code entry points:

- `src/cct-store/store.ts` — durable slot storage (revision CAS, fsync
  chain, `cct-store.lock`).
- `src/token-manager.ts` — the `TokenManager` class; owns lease lifecycle,
  rotation, proactive refresh, and usage fetch.
- `src/oauth/refresher.ts`, `src/oauth/usage.ts`, `src/oauth/scope-check.ts`,
  `src/oauth/header-parser.ts` — HTTP-layer helpers.
- `src/slack/cct/builder.ts`, `views.ts`, `actions.ts` — Block Kit card &
  modal surface.
- `src/slack/commands/cct-handler.ts`, `src/slack/command-parser.ts` —
  text-command grammar (READ-ONLY for token mutation).

## Token Slot Model

### Identity (slotId / name)

Every slot has an immutable `slotId` (ULID, generated by `ulid()` in
`TokenManager.addSlot`) and a mutable `name` (max 64 chars). All internal
maps, `Lease.ownerTag` suffixes, usage caches, and refresh dedupe keys are
keyed by `slotId` — renaming a slot does **not** invalidate in-flight
work. `name` is the only identifier surfaced in Slack UI (Block Kit and
text commands).

### Discriminated union (`setup_token` | `oauth_credentials`)

`TokenSlot` in `src/cct-store/types.ts` is a discriminated union on the
`kind` field:

```typescript
export type SlotKind = 'setup_token' | 'oauth_credentials';

export interface SetupTokenSlot {
  slotId: string;
  name: string;
  kind: 'setup_token';
  value: string; // sk-ant-oat01-…
  createdAt: string;
  acknowledgedConsumerTosRisk?: false;
}

export interface OAuthCredentialsSlot {
  slotId: string;
  name: string;
  kind: 'oauth_credentials';
  credentials: OAuthCredentials;
  createdAt: string;
  acknowledgedConsumerTosRisk: true;
}

export type TokenSlot = SetupTokenSlot | OAuthCredentialsSlot;
```

`setup_token` slots behave like the original single-env-var model — no
refresh, no usage API. `oauth_credentials` slots carry a `refreshToken`
and opt into proactive refresh + usage polling. The ToS-risk ack
(`acknowledgedConsumerTosRisk: true`) is required on construction for
`oauth_credentials` slots and enforced by `TokenManager.addSlot`.

### `SlotState`

Per-slot mutable state (keyed by `slotId` in the snapshot's `state` map):

```typescript
export interface SlotState {
  rateLimitedAt?: string;
  rateLimitSource?: 'response_header' | 'error_string' | 'manual';
  cooldownUntil?: string;
  authState: 'healthy' | 'refresh_failed' | 'revoked';
  activeLeases: Lease[];
  tombstoned?: boolean;
  lastUsageFetchedAt?: string;
  nextUsageFetchAllowedAt?: string;
  usage?: UsageSnapshot;
}
```

Rotation eligibility is computed by `isEligible(state, nowMs)` in
`token-manager.ts` — tombstoned, revoked, refresh_failed, and
still-cooling slots are excluded.

## Persistence

### `cct-store.json` schema

A single file under the soma-work data dir (`defaultCctStorePath()`
resolves to `<envPaths.data>/cct-store.json`). Snapshot shape:

```typescript
export interface CctStoreSnapshot {
  version: 1;
  revision: number;
  registry: { activeSlotId?: string; slots: TokenSlot[] };
  state: Record<string, SlotState>;
}
```

### Revision CAS

`CctStore.save(next)` fails with `CctStoreStaleWriteError` if
`next.revision !== current.revision + 1`. `CctStore.mutate(fn)` is the
correct all-purpose entry point: it loads the latest snapshot, runs `fn`
under the lock, bumps `revision`, and persists. Consumers that race will
each see a fresh `revision` before mutating.

### Durable fsync-chain writes

`CctStore.save` writes to a `*.tmp` sibling, `fsync`s that file, then
`rename()`s it over `cct-store.json`, then `fsync`s the parent directory.
This guarantees the post-crash on-disk state is either the pre-image or
the full post-image.

### Locking

`cct-store.lock` is the **only** cross-process lock. `CctStore.withLock(fn)`
is a bounded spin on `O_EXCL` with a hand-off file. The TokenManager uses
it for every mutation. Refresh and usage-fetch dedupe use **in-process**
`Map<slotId, Promise<…>>` only; HTTP calls happen outside the store lock
so we never hold the lock across the network. Lock ordering is strict:
caller → store lock → done.

## OAuth refresh

### Endpoint

`POST https://platform.claude.com/v1/oauth/token` (see
`src/oauth/refresher.ts`). Body:
`{ grant_type: "refresh_token", refresh_token, client_id }`.

### Client ID constant

`CLAUDE_OAUTH_CLIENT_ID = '9d1c250a-e61b-44d9-88ed-5944d1962f5e'` — the
hard-coded client ID shipped with the Claude Code CLI. Stored as a module
constant in `refresher.ts`.

### Proactive 7h buffer

`TokenManager.getValidAccessToken(slotId)` checks
`expiresAtMs - Date.now() < REFRESH_BUFFER_MS` (7 h) and refreshes
*before* handing out an access token. This prevents a race where a
long-running stream inherits a token that expires mid-call.

### In-process dedupe + cross-process lock

`TokenManager.refreshAccessToken(slot)` stores the in-flight Promise in
`this.refreshInFlight: Map<string, Promise<string>>`. The first caller
races the HTTP POST; every other caller awaits the same Promise. The
write-back (`store.mutate(…)`) takes the store lock — two processes
refreshing simultaneously are tolerated but the CAS ensures only one
committed revision wins, and the other retries on the next access.

### Token rotation (refresh_token reuse semantics)

Anthropic's OAuth server returns a new `refresh_token` on every
successful refresh; `refresher.ts` returns `OAuthCredentials` with the
rotated refresh token and the manager persists it atomically with the
new access token. If the next refresh returns 401, the manager sets
`authState = 'refresh_failed'`; 403 → `'revoked'`.

## Usage monitoring

### Endpoint

`GET https://api.anthropic.com/api/oauth/usage` (see
`src/oauth/usage.ts`, constant `CLAUDE_USAGE_URL`).

### `anthropic-beta` header

All usage requests include `anthropic-beta: oauth-2025-04-20` — the
endpoint is opt-in beta and will 404 without it.

### `five_hour` / `seven_day` / `seven_day_sonnet` schema

Response is parsed into the typed `UsageSnapshot`:

```typescript
export interface UsageWindow {
  utilization: number;
  resetsAt: string;
}

export interface UsageSnapshot {
  fetchedAt: string;
  fiveHour?: UsageWindow;
  sevenDay?: UsageWindow;
  sevenDaySonnet?: UsageWindow;
}
```

`utilization` is a 0..1 float on the wire; renderers
(`src/slack/cct/builder.ts::toPct`, `cct-handler.ts::toPctInt`) scale it
to an integer percentage and also accept 0..100 for forward-compat.

### Per-slot `lastFetchedAt` + `nextUsageFetchAllowedAt`

`TokenManager.fetchAndStoreUsage(slotId)` persists
`state.lastUsageFetchedAt` (from `snapshot.fetchedAt`) and
`state.nextUsageFetchAllowedAt` (server success = +2 min; failure = next
step on the backoff ladder).

### Backoff ladder: 2 → 5 → 10 → 15 (cap) minutes

`nextUsageBackoffMs(prev)` in `oauth/usage.ts` walks
`[2m, 5m, 10m, 15m]`. On every non-2xx, the manager calls
`applyUsageFailureBackoff(slotId)` which reads the current remaining
backoff and advances to the next step (capping at 15 min).

## Rate-limit detection

### Primary: response headers

`src/oauth/header-parser.ts::parseRateLimitHeaders` inspects
`anthropic-ratelimit-unified-5h-remaining`,
`anthropic-ratelimit-unified-7d-remaining`, and the matching `reset`
headers. When `remaining === 0`, the stream executor calls
`TokenManager.rotateOnRateLimit(reason, { source: 'response_header', … })`.

### Fallback: error string match

When headers are absent (older server build, network edge),
`parseCooldownTime(message)` (preserved from legacy) parses
`"resets 7pm"` / `"resets Apr 7, 7pm (Asia/Seoul)"` from the error body
and rotates with `source: 'error_string'`.

### `rateLimitSource` field on SlotState

The detected source (`response_header` | `error_string` | `manual`) is
persisted on `SlotState.rateLimitSource` so the Slack card can render a
provenance badge (see `buildSlotRow` in `src/slack/cct/builder.ts`).

## Rotation + Lease

### `pickNextHealthy` (exclude tombstoned / revoked / cooling / refresh_failed)

`TokenManager.rotateToNext` and `rotateOnRateLimit` both walk
`snap.registry.slots` starting at `currentIndex + 1`, skipping any slot
where `isEligible(state, now)` is false. If none is eligible they return
`null` and leave the active pointer untouched.

### Lease lifecycle

- `acquireLease(ownerTag, ttlMs = 15m)` — appends a `Lease` to the active
  slot's `activeLeases`. If the active slot is ineligible, rotates first.
- `heartbeatLease(leaseId)` — extends `expiresAt` by the original TTL.
  Throws on expired / unknown leases.
- `releaseLease(leaseId)` — removes from `activeLeases`.
- `reapExpiredLeases()` — purges expired leases and fully removes
  tombstoned slots with zero leases. Invoked by the reaper timer every
  `DEFAULT_REAPER_INTERVAL_MS` (30 s).

### Tombstone + drain removal

`removeSlot(slotId)` returns `{removed: true}` when the slot is leaseless,
else `{removed: false, pendingDrain: true}` with `state.tombstoned = true`.
The reaper finishes the delete once all leases have drained or expired.

## `authState`

- `healthy` — default; eligible for rotation.
- `refresh_failed` — last refresh attempt returned 401. Manager stops
  handing out tokens from this slot; operator must replace it. Surfaced
  as yellow-circle badge on the card.
- `revoked` — 403 from refresh or 403 from usage. The credential is
  considered invalidated and the slot is quarantined.

`markAuthState(slotId, state)` is the sole mutator; called from
`refreshAccessToken`'s catch block and from `fetchAndStoreUsage`'s 403
branch.

## Slack UI

### Card blocks (`builder.buildCctCardBlocks`)

`src/slack/cct/builder.ts` renders the header + one section/context pair
per slot + the action row (`Next / Add / Remove / Rename`) + a set-active
selector when `slots.length > 1`. `buildSlotRow` pulls `SlotState` and
emits an authState badge, a rate-limit timestamp (via
`formatRateLimitedAt`), 5h / 7d utilisation, cooldown hint, tombstone
marker, and lease count.

### Add modal

`buildAddSlotModal(selectedKind)` — split fields (name input, kind radio,
conditional value input). The kind radio has `dispatch_action: true` so
flipping `setup_token ↔ oauth_credentials` re-renders via `views.update`
with stable `block_id` / `action_id` keys so Slack preserves the user's
typed values. `oauth_credentials` adds a ToS-risk ack checkbox
(`CCT_BLOCK_IDS.add_tos_ack`).

### Remove modal

`buildRemoveSlotModal(slot, hasActiveLeases)` — warns on active leases
(slot will be tombstoned and drained). `private_metadata` carries the
`slotId` through `view_submission`.

### Rename modal

`buildRenameSlotModal(slot)` — single `plain_text_input` seeded with the
current name. `private_metadata` carries `slotId`.

### `block_id` / `action_id` stability across `views.update`

All modal field keys live in `src/slack/cct/views.ts` as the frozen
constants `CCT_BLOCK_IDS`, `CCT_ACTION_IDS`, `CCT_VIEW_IDS`. The kind-radio
flip re-uses the same keys so `state.values` survives the update.

### 3s ack budget, `response_action:errors` keyed by `block_id`

`src/slack/cct/actions.ts` acks within 3 s and returns
`response_action: 'errors'` with messages keyed by `block_id` on validation
failure — Slack highlights the offending field inline rather than closing
the modal.

## Timestamp format

The canonical rate-limit timestamp format is:

```
2026-04-18 13:42 KST / 03:42Z (5m ago)
```

Produced by `formatRateLimitedAt` in `src/util/format-rate-limited-at.ts`:
local wall clock in the user's IANA timezone (default `Asia/Seoul`), UTC
`HH:mmZ`, and a relative suffix (`now` / `Ns / Nm / Nh / Nd ago`).

Used by:

- `src/slack/cct/builder.ts::buildSlotRow` — per-slot context line.
- `src/slack/commands/cct-handler.ts::buildStatusTextFallback` — the
  plain-text fallback surfaced by `/z cct` when Block Kit is suppressed.

## Text command grammar (READ-ONLY)

Parsed by `src/slack/command-parser.ts::parseCctCommand`:

```
/z cct                  — show the Block Kit card
/z cct set <name>       — set the active slot
/z cct next             — rotate to next eligible slot
/z cct usage [<name>]   — show 5h / 7d usage snapshot for a slot (defaults to active)
```

Forbidden via text (handler emits an error pointing to the card):

```
/z cct add …            — open the Add modal from the /z cct card
/z cct rm … | remove …  — open the Remove modal from the /z cct card
```

Rationale: split-field entry + ToS-risk ack + scope validation + modal
dispatch semantics are too rich to encode in Slack text; the card is the
only supported mutation surface.

## ToS risk framing

Consumer Claude subscriptions grant tokens via the `claudeAiOauth` flow
for the CLI's own use. Programmatic reuse of these tokens for a bot
arguably violates Anthropic's consumer ToS. Every `oauth_credentials`
slot therefore requires `acknowledgedConsumerTosRisk: true` — enforced
both by the Add modal (checkbox) and `TokenManager.addSlot`. `setup_token`
slots (`sk-ant-oat01-…`) are issued for programmatic use and do not
require the ack.

## Secret redaction

`src/logger.ts::redactAnthropicSecrets` replaces matches of
`/sk-ant-(?:oat01|ort01|api03|admin01)-[A-Za-z0-9_-]+/g` with
`sk-ant-…-REDACTED` across every log path. `installConsoleRedaction`
wraps `console.{log,info,warn,error,debug}` so `console.*` callers also
get filtered. Used in `TokenManager.applyToken`, `addSlot`, and
`refreshAccessToken` logging.

## Migration

On first `CctStore.load()` the migrator (see `src/cct-store/migrate.ts`)
checks for a legacy `<dir>/token-cooldowns.json`. If present, it merges
each `{name, cooldownUntil}` entry into the new snapshot by `name` lookup
and renames the legacy file to `token-cooldowns.json.migrated.<ISO>` so
the migration is idempotent and auditable. `TokenManager.init` captures
legacy entries *before* the migrator renames the file so cooldowns can
be re-applied after env-based slot seeding.

## Backward compatibility

- `CLAUDE_CODE_OAUTH_TOKEN` — mirrored from the active slot's token
  value on every rotation / apply / lease acquire. Existing spawned
  processes that read this env var continue to work unchanged.
- `CLAUDE_CODE_OAUTH_TOKEN_LIST` — comma-separated list seeded into
  setup_token slots on first boot when the registry is empty.
  `name:value` and `name=value` forms both accepted; missing name falls
  back to `cct<N>`.
- `SOMA_CCT_DISABLE_ENV_SEED=true` — opt-out for test harnesses /
  containers that manage slots by other means.
- Legacy `/z cct set <name>` and `/z cct next` text commands still work
  unchanged — the slot model is invisible to operators who only ever use
  the text grammar.
